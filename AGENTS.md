# AGENTS.md

AAFinfo is a read-only command-line inspector for AAF (Advanced Authoring
Format) files. It produces a JSON report and a self-contained HTML report.
It never writes, modifies, or reaches the network.

For the full spec, see `PROJECT_OVERVIEW.md`.
For the phase-by-phase build plan, see `CODEX_HANDOFF.md`.
Read both before planning any work.

## Stack

- Language: Python 3.11+
- Package manager: `uv`
- Engine: `pyaaf2`, pinned to a fork at a specific commit (see `pyproject.toml`)
- Models: Pydantic v2 with `extra="forbid"`, schema-versioned
- CLI: Click (single command, no subcommands)
- Templating: Jinja2, autoescape on
- Tests: pytest

## Commands

- Install dev environment: `uv sync --dev --frozen`
- Run the CLI: `uv run aafinfo --help`
- Generate test fixtures: `uv run python examples/_generate.py`
- Run the test suite: `uv run pytest`
- Run the smoke flow: `uv run python examples/_smoke.py`

These commands may be run without asking. Anything that mutates external
state (publish to PyPI, push tags, open PRs, cut releases) requires explicit
instruction.

## Project Structure

- `src/aafinfo/` — application source
- `tests/` — pytest suite, mirrors `src/aafinfo/`
- `examples/` — fixture generator and smoke flow
- `.github/workflows/` — CI

The engine and model layers (`engine.py`, `models.py`) must remain importable
without Click, Jinja2, or filesystem side-effects. AAFpeek, a separate
forthcoming macOS app repo, imports them as a library.

## Code Style

- Type hints on all public functions and methods.
- `from __future__ import annotations` at the top of every module.
- Pydantic v2 models with `extra="forbid"` for any persisted shape. New
  fields require a `schema_version` bump and a CHANGELOG entry.
- `pathlib.Path` internally; `str` only at the CLI boundary.
- Long-form CLI flags only; no short aliases in 0.1.0.
- No star imports. Standard lib → third-party → local.

## Testing

- Run the full suite before declaring a phase complete.
- New code requires new tests.
- Fixtures are generated by `examples/_generate.py` using pyaaf2's write
  side. Never commit real-show AAFs to the repo.
- A failing test is a stop condition. Do not skip, comment out, or `xfail`
  it without explicit instruction.

## Git Workflow

- Branch naming: `phase-N-short-description` or `fix/short-name`.
- Commit messages: imperative mood, present tense ("Add X" not "Added X").
- One logical change per commit.
- Do not push, tag, or open PRs without explicit instruction.

## Boundaries

Never, without explicit instruction:

- Write to or modify the input AAF in any way (including touching its mtime).
- Make network calls from the runtime path. No telemetry, font fetching,
  analytics, or update checks.
- Include JavaScript in the HTML report. Filtering is server-side via
  `--filter`; the HTML must remain a single self-contained file.
- Subprocess out to `aaftool` or any other external tool.
- Add a PDF generation library. Browsers can print the HTML to PDF.
- Add macOS-only code paths or imports. AAFinfo is cross-platform.
- Read LibAAF source code while implementing AAFinfo. LibAAF is GPL2;
  format knowledge needed beyond what pyaaf2 provides belongs in upstream
  pyaaf2 PRs, not here.
- Add any dependency not listed in `PROJECT_OVERVIEW.md`.
- Edit `uv.lock` by hand.
- Rewrite history on shared branches.

## When in Doubt

If a real-world AAF causes `pyaaf2` to fail on some specific structure, do
**not** work around it inside AAFinfo. Record the failure in the report's
`warnings[]` array, keep going on the rest of the file, and add the case
to `TODO.md` for a future upstream pyaaf2 PR. The principle is:

> AAFinfo wraps pyaaf2; gaps in pyaaf2 are fixed in pyaaf2.

Anything that would constitute "AAFinfo's own AAF parser" violates this and
should be flagged for review before implementation.
